## Correctness track (plans 018–021, added 2026-07-04)

Source: a code review of a real Wazuh investigation transcript
(`T1048` on `wazuh.manager`, session `6cf3d31b-e78`) that never stopped
naturally and had to be `force_stop`'d after 4 rounds despite a fully
confident decision (H1 posterior = 1.0). Every finding below was confirmed by
reading the cited code, not inferred from the transcript alone.

- **018** (P0, M): the true root cause of the non-terminating loop. Any
  `ANTI_FORENSICS`-type obligation (triggered by low-trust/adversary-
  controllable evidence — common on real telemetry) can never be discharged
  automatically — its acceptance criterion checks two node attributes that
  are never written anywhere in the codebase. Since `open_hard()`
  unconditionally blocks stopping while any hard obligation is open, this is
  an unbounded deadlock. Fix: de-escalate (not discharge) the obligation from
  hard to VOI-gated once its SLA deadline passes, while keeping it visible in
  `unresolved_obligations` and adding a guardrail flag so the final report
  still reflects the coverage gap.
- **019** (P1, S): `get_voi_ranking` always raises (and swallows) a
  `TypeError` — it iterates a `CandidatePool` object directly and calls
  `len()` on it, but `CandidatePool` has neither `__iter__` nor `__len__`.
  Separately, `o_phase.py` destructively drains the same shared pool object
  referenced by `session.data["pool"]`, so even after fixing the iteration
  bug the tool would return empty results whenever called after `run_o_phase`
  — exactly when its own docstring recommends calling it. Both must be fixed
  together. No dependency on 018.
- **020** (P0, S): config-only. The pivot/seed_only/model-layer redesign for
  incremental multi-round chain discovery was only ever applied to
  `configs/engine.real_trace.yaml`; the default production path
  (`init_investigation(backend="soar_mcp")` with no explicit
  `TRACE_AGENT_ENGINE_CONFIG`) loads `configs/engine.yaml`, which still has
  none of it. A real run confirmed the effect: 5 of 6 attack-chain techniques
  were confirmed within Round 1 from 4 generic probes — the incremental-
  discovery design was never exercised. Fix: copy the missing sections
  verbatim into `engine.yaml`. No dependency on 018/019.
- **021** (P2, S): `get_obligation_status` doesn't surface the
  already-computed `Obligation.blocked_reason` field, so an agent (or human)
  investigating a stuck loop has no way to see *why* an obligation won't
  resolve short of reading source code. Purely additive fix. Recommended
  after 018 (which introduces the `"sla_expired_no_recovery_path"` value this
  tool should surface) but does not require it.

Recommended order: **020 and 018 first (both P0, independent of each other)
→ 019 → 021**. None of 018–021 depend on the demo-enablement track (014–017)
or the 009→013 convergence track.

## Second-round correctness track (plans 022–027, added 2026-07-04)

Source: a server-side reviewer's follow-up audit of a `real_trace_01`
investigation transcript, covering evidence-attachment quality, attack-graph
construction, and noise handling — a different pass than the 018–021 track
above (which came from a stop-condition/tooling audit of the same general
investigation). Every finding below was confirmed by reading the cited code
directly (see each plan's "Current state" section), not inferred from the
transcript alone.

- **022** (P0, M): the reviewer's top finding — two Wazuh informational
  events (`"Wazuh server started"`, log rotation) were ATTACHed to the
  leading hypothesis. Root cause is not a single explicit
  "posterior-saturation → default-attach" branch, but four compounding
  permissive defaults: an "unknown tactic → allow" fallback in
  `_tactic_can_follow`, a temporal window that grows to 24h as the graph
  grows, a flat structural-fit bonus given to any L1-attachable event
  regardless of which hypothesis is asking, and a lifecycle-hypothesis
  stage-fit bonus granted for *any* tactic. Also adds the missing L0 filter
  for Wazuh's own low-severity informational events, which currently has no
  code path anywhere in `IngestPipeline`.
- **023** (P1, M): generalizes part of Plan 018. Obligation materialization
  resolves target hosts against hosts *referenced by graph nodes*, not the
  investigation's actual allowed scope — so an obligation whose evidence
  lives on a VETOed out-of-scope host can still "successfully" materialize a
  probe, which VETO then discards, every round, for as long as the
  obligation stays open. Plan 018 bounds this to 3 rounds for
  `ANTI_FORENSICS` specifically via an SLA timer; this plan detects the
  unreachable case immediately, for all obligation types, since scope is
  knowable up front.
- **024** (P1, M): parent-edge selection for a newly-arrived event always
  takes the first candidate from an unordered list, with no timestamp
  comparison — so when two same-technique sibling events (e.g. two
  `T1110.001` failed-login attempts) are both graph frontier leaves, which
  one becomes a downstream event's "parent" is arbitrary, not
  chronological, and can produce a causally-backwards edge. Also clarifies
  (does not change the underlying math, which is already correct) a
  reporting gap where the routing-count metric most visible per round
  (`attach_bucket_count`) is easily mistaken for total graph growth, when
  `new_graph_nodes`/`graph_nodes` are the metrics that actually reconcile.
- **025** (P2, S): `real_trace_01`'s wide host-scoped `auth_log` probes
  return every `sshd`/`pam` event for that host from the shared Wazuh
  Indexer — both the scenario's injected noise *and* its injected attack
  events *and* potentially unrelated real syslog — with nothing to
  distinguish them once they pass triage's required-field check (all of
  them have *some* technique). Adds an opt-in, scenario-configured MITRE
  technique allow-list so out-of-chain events get flagged as scope-noise for
  reporting purposes without changing ATTACH/WEAK/PARK routing itself.
- **026** (P2, S): the runtime posterior update has no minimum-probability
  floor (unlike the seed-time null-anchor bounds in `belief.py`, which only
  apply once at the start). Many rounds of strong-attack evidence against
  `__null__` can drive its log-posterior arbitrarily low — the reviewer's
  cited `2.9e-18` is consistent with exactly this, not a one-off glitch.
  Adds a floor consistent with the `1e-6` epsilon already used elsewhere in
  the same class.
- **027** (P2, S): three of the four shared-Indexer mock scenarios already
  have query-level isolation (`incident_id`/`is_attack` tagging via
  `scenario_registry.py`); `real_trace_01` has none, by design (it has no
  `incident_id` field). Wires the rule-group tag its own custom Wazuh rules
  already emit (`rule.groups: real_trace`, documented in
  `REAL_TRACE_SCENARIO.md` §8) into the same isolation mechanism as an
  additional, independent scope field — reducing cross-scenario pollution
  without requiring an `incident_id` this scenario intentionally doesn't
  have.

Recommended order: **022 first (P0, highest-confidence finding, no
dependencies)**, then **023 and 024 (both P1, independent of each other and
of 022)**, then **025 → 027 (025 must land first; 027 is complementary, not
blocking)**, and **026 independently at any point (numerics-only, no
functional dependency on the others)**. None of 022–027 depend on the
demo-enablement track (014–017) or the 009→013 convergence track. 023 is
complementary to (not a duplicate of) 018.

  exposed as calibrated probabilities.

- Plan 010 removes duplicate setup/report paths and preserves real stop reasons.
- Plan 011 centralizes phase transitions, failures, and streaming contracts.

  compatibility implementation.
- Plan 013 uses the canonical lifecycle so session eviction and model cleanup
  have one owner.

## Findings considered and rejected

- Replace all fixed thresholds with an LLM: rejected. Evidence integrity,
  permissions, budgets, probability calibration, and stopping safety require
  deterministic or statistically calibrated mechanisms.
- Let the model write SessionGraph directly: rejected. Models may propose
  typed intents; schema validation and K-phase code remain the only writers.
- Treat model self-reported confidence as probability: rejected. Confidence
  remains unavailable until calibrated against independent labels.
- Keep the standalone `demo/` backend as a second runtime: rejected. Its files
  are already staged for deletion in the current worktree; replay and
  presentation should use the canonical Deep Agent UI and versioned phase
  fixtures.
- Merge `trace_engine` and `trace_agent` into one package: rejected. Preserve
  the domain-core/application boundary; remove duplicated behavior across it.
